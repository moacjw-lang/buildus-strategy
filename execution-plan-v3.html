빌드어스 플랫폼

콘텐츠

"사람들을 데려오고, 전문성을 증명하는 것"

유튜브 (가이드 트랙 + 데이터 트랙) · 아파트별 비용 SEO 페이지 · 블로그/웹 콘텐츠

지금은 판매/매출이 목적이 아니다.
플랫폼의 기반이 될 데이터를 쌓고, 콘텐츠로 사람들이 찾아올 이유를 만드는 단계.
02 각자 뭘 갖고 있는가
누가 뭘 만들 수 있는지가 실행 순서를 결정한다.

대표  인테리어 전문 지식

  • 16개 공정 체크리스트 (400+ 항목, 이미 정리됨)
  • 공정별 시공 순서/방법 — 현장 경험 5년치
  • 자재-공정 매핑 — "방수에 뭐가 필요한지" 알고 있음
  • 호환성 지식 — "이 본드 위에 이 타일 안 됨" 등
  • 도면/공정표 — 실제 프로젝트 기반
  • 아파트별 비용 감각 — 경험 기반 추정
  • 얼굴/목소리 — 유튜브 출연

유통  자재 + 가격 + 물류

  • 자재 카탈로그 — 품명, 규격, 브랜드, 제조사
  • 도매가/업체가 — 실제 매입 단가
  • 소비자가 갭 — 인터넷가 대비 얼마나 싼지
  • 공급사/제조사 네트워크 — 입점 협상 가능
  • 배송 인프라 — 택배/화물 배송 체계
  • 샘플 확보 능력 — 타일/마루/페인트 실물

개발자  기술 구현력

  • 웹/앱 풀스택
  • AI/RAG 파이프라인
  • 이커머스 시스템
  • 데이터 시각화
  • 위 3명의 데이터를 제품으로 변환하는 사람
  • 단, 혼자이므로 한 번에 하나만

어시  범용 실행력

  • 데이터 정리/입력
  • 인터넷 조사 — 경쟁사 가격, SEO 키워드
  • 콘텐츠 작성 — 상품 설명, SEO 글
  • CS 대응
  • 핵심: 대표/유통의 머릿속을 문서화

4인 팀이라 자원이 적다. 한 번 만든 데이터가 여러 곳에서 재사용되어야 효율이 나온다.
그래서 데이터를 처음부터 구조화된 형태(서식에 맞춰)로 만들어야 한다. 나중에 다시 정리하는 건 이중 작업.
04 플랫폼에 들어갈 데이터
플랫폼을 만들기 전에 준비해야 하는 "원재료". 누가 만들 수 있는지가 정해져 있다.
데이터내용만드는 사람쓰이는 곳
자재 카탈로그 + 가격 품명, 규격, 브랜드, 도매가, 업체가 유통 마켓플레이스, 비용 분석기, 가격 비교
공정별 필요 자재 "방수 공정 = 방수액 + 부직포 + 롤러 + ..." 대표 비용 분석기, 수량 계산기, 챗봇
공정별 시공 가이드 시공 순서, 주의사항, 양생 시간, 실패 포인트 대표 가이드 페이지, 유튜브, 챗봇, 상세페이지
자재 호환성 "이 방수재 위에 이 본드 OK/NG" 대표 호환성 체커, 마켓플레이스, 챗봇
아파트별 비용 추정 "래미안 25평 욕실 방수 = 약 OO만원" 대표 SEO 랜딩, 비용 분석기
인터넷 최저가 쿠팡/네이버 기준 자재별 소비자가 어시 가격 투명성 UI
상품 사진 + 설명 자재 실물 사진, 상세 설명 텍스트 어시 + 유통 마켓플레이스 상품 페이지
공정표/도면 표준 공정 순서, 공정간 의존성, 예상 일정 대표 프로젝트 관리 도구
체크리스트 Q&A 변환 기존 체크리스트를 "질문-답변" 쌍으로 재구성 어시 (대표 감수) AI 구매 어시스턴트 RAG
이 데이터의 80%는 대표와 유통의 머릿속에만 있다.
문서로 뽑아내는 게 가장 먼저 할 일이고, 가장 오래 걸리는 일이다.
어시 = 이 두 사람 옆에서 받아적고, 빈 곳은 조사해서 채우는 사람.
05 실행 순서
데이터가 먼저, 개발이 다음. 콘텐츠는 데이터와 동시에 나온다.
1

데이터 추출 + 시스템 설계

대표와 유통의 머릿속을 문서로 꺼낸다. 개발자는 시스템을 설계한다.

STEP 1 내부 순서: 동시에 다 하지 않는다. A → B → C 순서가 있다.
뒤 단계의 입력값이 앞 단계의 산출물에 의존하기 때문이다.
PHASE A — 범위 설정 + 기초 데이터
대표
☐
우선 공정 5개 선정 — 검색량/수요 기반으로 먼저 다룰 공정 결정
추천: 방수, 도장, 타일, 도배, 마루 (셀프 수요 상위)
유통
☐
취급 자재 전체 목록 초안 — 품명, 규격, 브랜드, 제조사, 도매가, 업체가
산출물: 자재 카탈로그 초안 (전체 취급 자재)
개발자
☐
기술 스택 결정 + DB 스키마 설계 — 자재 DB 구조, 이커머스 아키텍처
산출물: 기술 설계 문서
어시
☐
유통 보조 — 카탈로그 데이터 입력/정리
유통이 말하는 걸 구조화된 형태로 입력

▼ Phase A가 끝나면: 대표가 어떤 공정을 먼저 할지 정해짐 + 유통의 자재 목록 초안이 있음 → Phase B 시작 가능

PHASE B — 핵심 데이터 추출
대표+ 어시 동행
☐

☐
5개 공정 × 필요 자재 매핑 — 유통 카탈로그를 참조하면서 작성

산출물: 호환성 테이블 (= 마켓플레이스 경고 + 호환성 체커)
유통
☐
카탈로그 완성 — 초안 보완 + 가격 정보 확정
산출물: 전체 자재 가격 테이블 (마켓플레이스 + 비용 분석기의 가격 엔진)
☐
공급사/제조사 정보 정리 — 입점 가능 업체, 리드타임, 최소 주문 수량
산출물: 공급사 DB (마켓플레이스 운영 기반)
개발자
☐
와이어프레임 — 대표와 핵심 UX 플로우 확정
산출물: 화면 설계서 (비용 분석기, 마켓플레이스, 상세페이지)
어시
☐
대표 옆에서 문서화 — 가이드, 매핑, 호환성 데이터를 서식에 맞춰 정리
대표가 말하면 어시가 기록하고 구조화

▼ Phase B가 끝나면: 5개 공정의 가이드 + 자재 매핑 + 호환성 + 전체 가격 데이터 완성 → 개발 시작 가능 + 유튜브 촬영 가능

PHASE C — 보조 데이터 + 콘텐츠 시작 + 개발 착수
대표

산출물: 아파트별 비용 데이터 (SEO 랜딩 + 비용 분석기)

유튜브 촬영 시작 — Phase B에서 쓴 가이드 5편 기반

어시
☐

산출물: 가격 비교 테이블 (가격 투명성 UI)
☐

☐

유통
☐
배송 프로세스 설계 — 택배/화물 배송 체계, 포장 기준
산출물: 배송 운영 매뉴얼 (마켓플레이스 오픈 전 준비)
☐
샘플 확보 — 타일/마루/페인트 실물 샘플 준비
산출물: 샘플 재고 (샘플 배송 서비스 대비)
개발자
☐
마켓플레이스 개발 시작 — Phase B의 데이터가 들어가는 제품
STEP 2로 이어짐

- 5개 공정의 가이드 + 자재 매핑 + 호환성 데이터 완성

- 개발에 필요한 데이터 원재료 대부분 준비
2

마켓플레이스 MVP + 콘텐츠 확대

개발자는 마켓플레이스를 완성한다. 나머지는 데이터를 확장하고 콘텐츠를 만든다.

☐

☐

- 빌드어스 마켓플레이스 오픈 (자재 구매 가능)

- 16개 공정 데이터 대부분 완성 (or 진행 중)
3

유입 도구 개발

4

전환 + 리텐션 강화

오는 사람을 더 많이 사게 하고, 한번 온 사람을 붙잡는다.

06 데이터 의존성
누구의 산출물이 누구에게 가는지. 이게 꼬이면 전체가 멈춘다.

대표의 산출물이 가는 곳

대표: 시공 가이드 5편 → 개발자: 상품 상세페이지 콘텐츠
대표: 시공 가이드 5편 → 유튜브 대본 (바로 촬영)
대표: 시공 가이드 5편 → AI 챗봇 RAG 지식 소스
대표: 공정-자재 매핑 → 개발자: 비용 분석기 엔진
대표: 호환성 데이터 → 개발자: 호환성 체커 + 마켓플레이스 경고
대표: 아파트 비용 추정 → 개발자: SEO 랜딩 데이터 + 비용 분석기

유통의 산출물이 가는 곳

유통: 자재 카탈로그 + 가격 → 개발자: 마켓플레이스 상품 DB
유통: 자재 카탈로그 + 가격 → 개발자: 비용 분석기 가격 엔진
유통: 자재 카탈로그 + 가격 → 개발자: 가격 투명성 UI
유통: 공급사 정보 → 마켓플레이스 입점/배송 운영

어시의 산출물이 가는 곳

어시: 인터넷 최저가 조사 → 개발자: 가격 투명성 UI ("OO원 절약")
어시: Q&A 변환 (STEP 3) → 개발자: AI 챗봇 RAG 소스 (STEP 4)
어시: 상품 사진/설명 → 마켓플레이스 상품 페이지
어시: SEO 콘텐츠 → 아파트별 비용 블로그/웹 페이지

상호 의존

대표: 공정-자재 매핑 ↔ 유통: 카탈로그
대표가 "방수에 이 자재"라고 하려면 유통이 뭘 취급하는지 알아야 하고,
유통이 자재를 분류하려면 대표가 어떤 공정에서 쓰이는지 알려줘야 한다.
→ Phase B에서 둘이 함께 작업해야 하는 이유.
병목 포인트:
대표가 가이드와 자재 매핑을 안 만들면 → 개발자가 비용 분석기/상세페이지를 못 만듦
유통이 카탈로그를 안 주면 → 개발자가 마켓플레이스를 못 만듦

결론: STEP 1 Phase B에서 대표+유통+어시가 같이 앉아서 데이터를 뽑는 게 전체 속도를 결정.
07 개발 순서
개발자 1명. 동시에 두 개 못 만든다. 이 순서를 지킨다.
순서만드는 것필요한 데이터이유
1 마켓플레이스 유통 카탈로그 + 가격
대표 가이드
어시 사진/설명 + 인터넷가 		플랫폼의 중심. 이게 없으면 나머지 전부 연결할 곳이 없음
2 비용 분석기 + 수량 계산 대표 공정-자재 매핑 + 스타일 매핑
유통 가격 		최대 유입 도구. "절약 금액" → 마켓플레이스 구매 연결
3 아파트별 비용 + 견적 비교기 대표 비용 추정
어시 SEO 콘텐츠 		SEO 유입 확대. 정적 페이지라 상대적으로 빠름
4 AI 구매 어시스턴트 어시 Q&A 데이터
대표 감수/검증 		전환율 부스터. 상품이 먼저 있어야 추천 가능
5 호환성 체커 + 샘플 온라인화 대표 호환성 데이터
유통 샘플 재고 		구매 확신 강화. "실패 안 함" 메시지의 도구 버전
6 프로젝트 관리 + UGC + 레벨 대표 공정표 + 체크리스트 리텐션. 사용자가 모인 후에 의미 있음
08 어시스턴트 배치 전략
어시는 유일한 "범용 자원". 어디에 붙이느냐가 전체 효율을 좌우한다.
STEP어시 배치이유
STEP 1
Phase A		 유통 100%
카탈로그 데이터 입력/정리 		Phase A에서 대표는 공정 선정만 하므로 보조 불필요.
유통의 카탈로그 작업이 양이 많음.
STEP 1
Phase B		 대표 80% + 유통 20%
- 대표 옆: 가이드 받아적기, 호환성 표 정리, 매핑 테이블 작성
- 유통: 카탈로그 잔여 정리 		Phase B의 핵심은 대표의 머릿속을 문서화하는 것.
이게 가장 시간이 많이 걸리고 어시 도움이 가장 필요한 구간.
STEP 1
Phase C		 독립 작업 100%
인터넷 최저가 조사 + SEO 키워드 + 상품 사진/설명 		대표는 유튜브 촬영, 유통은 배송 설계 — 어시는 독립 리서치 업무
STEP 2 대표 50% + SEO 50%
- 대표 보조: 나머지 공정 가이드 문서화
- SEO: 아파트별 비용 콘텐츠 작성 		대표는 6~16번 공정 데이터를 계속 만듦 → 어시 보조 필요.
동시에 SEO 콘텐츠도 써야 함.
STEP 3 Q&A 변환 50% + CS 30% + 데이터 20%
- 체크리스트 Q&A 변환 (챗봇 준비)
- CS 대응
- 가격 데이터 주기적 업데이트 		STEP 4 챗봇 직전 → Q&A 데이터 준비 시작.
마켓플레이스 운영 중 → CS 발생.
STEP 4 CS 40% + 챗봇 30% + 커뮤니티 30%
- CS 패턴 → FAQ → 챗봇 보강
- 커뮤니티 관리 		CS 패턴이 축적됨 → 챗봇 자동화로 CS 부담 줄이기

정리

이 팀의 효율은 "하나의 데이터가 여러 곳에서 쓰이는 구조"에서 나온다.

대표가 가이드를 쓰면 → 플랫폼 콘텐츠 + 유튜브 대본 + 챗봇 지식
유통이 카탈로그를 만들면 → 마켓플레이스 + 비용 분석기 + 가격 비교
어시가 이 과정을 문서화하고, 빈 곳은 조사해서 채운다
개발자가 이 데이터를 제품으로 변환한다

STEP 1이 모든 것의 기반.
Phase A → B → C 순서를 지키고,
데이터를 처음부터 구조화된 서식에 맞춰 만들면
나중에 개발자가 받아서 바로 넣을 수 있다.

그 서식이 다음에 만들 "서식 폼"이다.

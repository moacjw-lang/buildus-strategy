Execution Plan v2

빌드어스 실행 계획

각 팀원이 가진 자원 기준 · 2026-03-29
01 각자 뭘 갖고 있는가
실행 계획은 "뭘 할까"가 아니라 "누가 뭘 갖고 있는지"에서 시작한다.

대표  인테리어 전문 지식

  • 16개 공정 체크리스트 (400+ 항목, 이미 정리됨)
  • 공정별 시공 순서/방법 — 머릿속에 있는 현장 경험 5년치
  • 자재-공정 매핑 — "방수에 뭐가 필요한지" 알고 있음
  • 호환성 지식 — "이 본드 위에 이 타일 안 됨" 등
  • 도면/공정표 — 실제 프로젝트 기반 정리 가능
  • 아파트별 비용 감각 — 현장 경험 기반 추정
  • 얼굴/목소리 — 유튜브 출연 가능
  • 키트 구성 판단 — "방수에 이것들이 한 세트" 결정권

유통  자재 + 가격 + 물류

  • 자재 카탈로그 — 품명, 규격, 브랜드, 제조사
  • 도매가/업체가 — 실제 매입 단가
  • 소비자가 갭 — 인터넷가 대비 얼마나 싼지
  • 공급사/제조사 네트워크 — 입점 협상 가능
  • 배송 인프라 — 택배/화물 배송 체계
  • 재고 관리 노하우
  • 샘플 확보 능력 — 타일/마루/페인트 실물
  • 키트 원가 계산 — 마진 구조 세팅

개발자  기술 구현력

  • 웹/앱 개발 — 혼자 풀스택으로 만듦
  • AI/RAG 파이프라인 — 챗봇 구축 가능
  • 이커머스 시스템 — 결제, 주문, 배송 연동
  • 데이터 시각화 — 비용 분석 리포트 등
  • 위 3명이 만든 데이터를 제품으로 변환하는 사람
  • 단, 혼자이므로 한 번에 하나만 가능

어시  범용 실행력

  • 데이터 정리/입력 — 엑셀, 노션, 어디든
  • 인터넷 조사 — 경쟁사 가격, SEO 키워드 등
  • 콘텐츠 작성 — 상품 설명, 블로그, SEO 글
  • CS 대응 — 주문/배송/자재 문의
  • 잡무 전반 — 사진 촬영, 패키징 보조 등
  • 핵심: 대표/유통의 머릿속 지식을 문서화하는 역할
이게 왜 중요한가:
개발자는 빈 플랫폼을 만드는 게 아니라, 대표의 전문 지식 + 유통의 자재 데이터가 들어간 플랫폼을 만드는 것이다. 데이터가 먼저 준비되지 않으면 개발자는 껍데기만 만들게 됨.
02 플랫폼에 들어갈 데이터
개발 전에 만들어야 하는 "원재료". 누가 만들 수 있는지가 정해져 있다.
데이터내용만들 수 있는 사람들어가는 곳
자재 목록 + 가격 품명, 규격, 브랜드, 도매가, 업체가, 소비자가 유통 마켓플레이스, 비용 분석기
공정별 필요 자재 "방수 공정에는 방수액 + 부직포 + 롤러 + ..." 대표 키트 구성, 비용 분석기, 챗봇
자재 호환성 "이 방수재 위에 이 본드 OK/NG" 대표 마켓플레이스, 챗봇
공정별 시공 가이드 시공 순서, 주의사항, 양생 시간, 실패 포인트 대표 가이드 페이지, 챗봇, 유튜브
키트 구성 + 원가 5종 키트 구성품, 원가, 판매가, 마진 대표 + 유통 마켓플레이스 (킬러 상품)
아파트별 비용 추정 "래미안 25평 욕실 방수 = 약 OO만원" 대표 SEO 랜딩, 비용 분석기
인터넷 최저가 조사 쿠팡/네이버 기준 자재별 소비자가 어시 가격 투명성 UI ("OO원 절약")
체크리스트 Q&A 변환 기존 체크리스트를 "질문-답변" 쌍으로 재구성 어시 (대표 감수) AI 챗봇 RAG 소스
상품 사진 + 설명 자재 실물 사진, 상세 설명 텍스트 어시 + 유통 마켓플레이스, 스마트스토어
공정표/도면 자료 표준 공정 순서, 공정간 의존성, 예상 일정 대표 프로젝트 관리 도구, 공정 알림
현실: 이 데이터의 80%는 대표와 유통의 머릿속에만 있다. 문서로 뽑아내는 게 가장 먼저 할 일이고, 가장 오래 걸리는 일이다. 어시스턴트의 핵심 역할 = 이 두 사람 옆에 붙어서 받아적는 것.
03 실행 순서
데이터가 먼저, 개발이 다음. 개발 없이 팔 수 있는 건 바로 판다.
1

데이터 뽑기 + 즉시 판매

대표와 유통의 머릿속을 문서로 꺼낸다. 동시에 스마트스토어로 먼저 판다.

대표— 현장 지식 문서화
☐
키트 5종 구성 확정 — "방수에는 이것들이 한 세트다" 결정
산출물: 키트별 구성품 리스트 → 유통에게 전달
☐
공정별 필요 자재 매핑 — 16개 공정 × 필요 자재 목록
산출물: 공정-자재 매핑 테이블 → 비용 분석기의 핵심 엔진
☐
자재 호환성 데이터 — 주요 자재 50종 간 OK/NG 매트릭스
산출물: 호환성 테이블 → 마켓플레이스 경고 시스템
☐
공정별 시공 가이드 원고 — 순서, 주의사항, 실패 포인트
산출물: 가이드 텍스트 16편 → 상품 상세페이지 + 유튜브 대본
☐
아파트별 비용 추정 — 경험 기반 50개 단지
산출물: 단지별 비용 데이터 → SEO 랜딩 + 비용 분석기
유통— 자재 데이터 + 즉시 판매
☐
TOP 50 자재 목록 + 가격 정리 — 품명, 규격, 도매가, 업체가
산출물: 자재 가격 테이블 → 마켓플레이스 + 비용 분석기
☐
키트 5종 원가 계산 + 제작 — 대표에게 구성품 받고, 원가/마진 세팅, 물리적 패키징
산출물: 판매 가능한 키트 5종
☐
네이버 스마트스토어 오픈 — 키트 5종 + 주요 개별 자재 등록, 바로 판매 시작
산출물: 매출 시작 (개발 완료 안 기다림)
☐
샘플 재고 확보 — 타일/마루/페인트 실물 샘플 + 배송 프로세스
산출물: 카톡으로 샘플 신청 받기 가능
어시— 대표/유통 옆에서 문서화
☐
인터넷 최저가 조사 — TOP 50 자재의 쿠팡/네이버 소비자가 수집

체크리스트 → Q&A 변환 — 기존 16개 체크리스트를 "질문-답변" 쌍으로 재구성
산출물: 챗봇 RAG 소스 데이터
☐
상품 사진 촬영 + 설명 작성 — 유통이 가져다주는 자재 실물
산출물: 스마트스토어 + 마켓플레이스 상품 정보
☐
SEO 키워드 50개 조사 — 검색량, 경쟁도, 빌드어스가 노릴 수 있는 것
산출물: SEO 전략 키워드 리스트

- 개발자가 쓸 데이터 원재료 대부분 준비 완료
- 개발 시작 준비 OK

04 데이터 의존성
누구의 산출물이 누구에게 가는지. 이게 꼬이면 전체가 멈춘다.

대표의 산출물이 가는 곳

대표: 키트 구성 확정 → 유통: 키트 제작 + 원가 → 스마트스토어 판매
대표: 공정-자재 매핑 → 개발자: 비용 분석기 엔진
대표: 시공 가이드 원고 → 개발자: 상품 상세페이지 콘텐츠
대표: 시공 가이드 원고 → 유튜브 대본
대표: 아파트 비용 추정 → 개발자: SEO 랜딩 페이지 데이터

유통의 산출물이 가는 곳

유통: 자재 목록 + 가격 → 개발자: 마켓플레이스 상품 DB
유통: 자재 목록 + 가격 → 개발자: 비용 분석기 가격 엔진
유통: 키트 원가/마진 → 개발자: 키트 상품 등록
유통: 키트 완성품 → 스마트스토어 즉시 판매

어시의 산출물이 가는 곳

어시: 인터넷 최저가 조사 → 개발자: 가격 투명성 UI ("OO원 절약")
어시: 체크리스트 Q&A 변환 → 개발자: AI 챗봇 RAG 소스
어시: 상품 사진/설명 → 스마트스토어 + 마켓플레이스
병목 포인트:
대표가 공정-자재 매핑과 가이드 원고를 안 만들면 → 개발자가 비용 분석기/상세페이지를 못 만듦
유통이 가격 데이터를 안 주면 → 개발자가 마켓플레이스를 못 만듦

결론: STEP 1에서 대표와 유통이 데이터를 빨리 뽑는 게 전체 일정을 결정한다. 어시는 이 두 사람에게 붙어서 받아적고, 정리하고, 빈 곳을 조사해서 채운다.
05 개발 순서 (혼자이므로 순서가 생명)
개발자 1명. 동시에 두 개 못 만든다. 이 순서를 지킨다.
순서만드는 것필요한 데이터 (누가 주는가)이유
1 마켓플레이스 유통 자재 목록 + 가격
대표 가이드 원고
어시 사진/설명 + 인터넷가 		수익 직결. 이게 없으면 나머지 전부 허공에 짓는 것
2 비용 분석기 + 수량계산 대표 공정-자재 매핑 + 아파트 데이터
유통 자재 가격 		최대 유입 도구. "바로 구매" 연결이 가능해진 상태
3 아파트별 비용 + 견적 비교기 대표 비용 추정 데이터 SEO + TAM 확장. 정적 페이지라 빠름
4 AI 구매 어시스턴트 어시 Q&A 쌍 데이터
대표 감수/검증 		전환율 부스터. 상품이 먼저 있어야 추천 가능
5 프로젝트 관리 + 레벨 대표 공정표 + 체크리스트 리텐션. 사용자가 모인 후
06 어시스턴트 배치 전략
어시는 유일한 "범용 자원". 어디에 붙이느냐가 전체 속도를 좌우한다.
STEP어시 배치이유
STEP 1 대표 50% + 유통 50%
- 대표 옆: 가이드 받아적기, 호환성 표 정리, 아파트 비용 조사 보조
- 유통 옆: 인터넷가 조사, 상품 사진/설명, 스마트스토어 등록 보조 		데이터 뽑기가 전체 일정을 결정. 두 사람의 병목을 줄이는 게 최우선
STEP 2 SEO 콘텐츠 60% + CS 40%
- 아파트별 비용 블로그/웹 콘텐츠 작성
- 스마트스토어 CS 대응 		대표/유통은 데이터 뽑기 끝남. 어시는 콘텐츠 생산 + 운영으로 전환
STEP 3 CS 60% + 데이터 업데이트 40%
- 주문/배송/자재 CS 대응
- 자재 가격 주기적 업데이트, 신규 자재 등록 		매출 증가 → CS 증가. 가격 데이터도 최신 유지 필요
STEP 4 CS 40% + 챗봇 보강 30% + 커뮤니티 30%
- FAQ → 챗봇 데이터 추가
- 커뮤니티 스팸/분쟁 관리 		CS 패턴이 축적됨 → 챗봇 자동화로 CS 부담 줄이기
07 개발 전 매출이 나는 구조
자사 플랫폼이 없어도 팔 수 있다. 유통 담당이 있으니까.
채널판매 방법누가 만드는가
네이버 스마트스토어 키트 5종 + 주요 개별 자재 유통 셋업 + 운영 / 어시 사진/설명
유튜브 설명란 "이 영상에 사용된 자재" → 스마트스토어 링크 대표 콘텐츠
카톡 수동 주문 샘플 신청 → 실물 확인 → 본품 주문 유통 운영
셀인카페 "공정별 자재 키트 만들었습니다" 소개 글 대표 / 어시
이 구조가 가능한 이유: 유통 담당이 있기 때문.
3명 팀이었으면 대표가 키트 제작 + 배송 + 유튜브를 혼자 해야 했음 → 병목.
유통 담당이 물류/판매를 독립 운영하니까, 대표는 데이터 정리 + 콘텐츠에만 집중.

정리

결국 이 사업의 속도는 "데이터를 얼마나 빨리 뽑느냐"에 달렸다.

대표의 머릿속 → 공정 가이드, 자재 매핑, 호환성, 비용 추정
유통의 머릿속 → 자재 목록, 가격, 키트 원가, 공급사
어시 → 위 두 사람 옆에서 받아적고, 빈 곳은 조사해서 채운다
개발자 → 위 데이터가 오면 제품으로 변환한다

그리고 개발을 기다리지 않는다.
스마트스토어에서 키트 먼저 팔고, 유튜브 먼저 시작하고, 샘플 먼저 보낸다.
자사 플랫폼은 이미 굴러가는 사업을 더 효율적으로 만드는 도구일 뿐이다.
